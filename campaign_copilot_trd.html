Callback loop (the system design showcase)

1
CRM sends campaign
POST /send to channel service with recipient list, message, channel
2
Channel service queues per-recipient
Each message gets a unique message_id. Returns 202 immediately (non-blocking).
3
Async delivery simulation
setTimeout jitter (0.5–3s). ~90% delivered, ~8% failed, ~2% unknown. Opened/read events fire after delivery with additional jitter.
4
Callback POST to CRM
POST /api/receipts on CRM with {message_id, campaign_id, status, timestamp}. Retries 3× on 5xx with exponential backoff.
5
CRM updates DB + emits SSE
Upserts message status, recalculates campaign stats, broadcasts to all connected SSE clients.
6
Dashboard updates live
Sent / Delivered / Failed / Opened counters tick up in real time. No polling.

Key tradeoffs (state what you chose and why)

SSE over WebSockets — SSE is unidirectional (server → client), which is all we need for delivery events. No WS handshake, no reconnect complexity. At this scale, perfect.
No job queue (BullMQ/Redis) — Simulated callbacks use in-process setTimeout. At real scale you'd use a queue. Explicitly acknowledged: "I'd use BullMQ + Redis at scale, but setTimeout works for this demo and keeps the infra simple."
Supabase over local PostgreSQL — No infra to manage, free tier sufficient, instant backups. Tradeoff: vendor dependency.
Full conversation history per API call — Gemini has no memory between calls. We pass the entire chat history each turn. Tradeoff: token usage grows per conversation. Acceptable for demo length.
No auth — Single-tenant demo. Explicitly cut. "Multi-tenant auth would require JWT + row-level security in Supabase — out of scope for this assignment."
